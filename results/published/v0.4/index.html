Measured finding

Loops can help, do nothing, or regress

Judge each loop against Direct within the same model. More calls are useful only when rescued outcomes exceed regressions at an acceptable cost.

Design limitation

Goal-based looping cannot create missing capability

The controller adds a deterministic public-test goal, a fixed verification skill, and at most five Maker attempts. Extra attempts count as test-time scaling and are useful only when the model can convert visible failure evidence into a better edit.

Inference boundary

Qualification, not a general coding leaderboard

The result applies to this small deterministic repair suite. It does not establish broad model quality or serving efficiency.

LOOP DESIGN

Controller flow

The nodes below match the tested controller

Direct baseline

One Maker attempt; no controller retry
  1. Reset the pinned clean task worktree
  2. Maker returns full-file edit JSON
  3. Validate, apply, and run public tests once
  4. Run isolated hidden evaluation after the episode

Bounded Retry

At most three evidence-driven Maker attempts
  1. Start from the same agent-visible task bundle
  2. On failure, return sanitized public evidence
  3. Repair the current visible worktree
  4. Exit on public pass or the three-attempt cap
  5. Run isolated hidden evaluation after the episode

Goal Skill Loop

Deterministic goal, fixed skill, at most five attempts
  1. State public-test pass as the deterministic goal
  2. Apply the same five-part verification skill every turn
  3. Return sanitized public evidence after failures
  4. Exit on public pass or the five-attempt cap
  5. Count every logical token and evaluate hidden tests only afterward
Evaluation boundaryPublic tests are the only agent-visible stop condition. Hidden tests, gold edits, evaluator paths, and hidden outcomes are never returned to the model.
